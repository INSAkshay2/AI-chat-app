# MASTER PIPELINE PROMPT
# Chain-of-Evidence Claim Verification System
# Version 1.0

## System purpose

This is a 4-pass multimodal pipeline that evaluates insurance and logistics damage claims.
Each pass has a single, isolated job. Passes run in order: 1 → 2 → 3 → 4.
Pass 1 runs in parallel across all images for a claim.

---

## Pipeline overview

PASS 1: Image Forensics       → See: prompts/p1_image_forensics.txt
PASS 2: Claim Decomposition   → See: prompts/p2_claim_decompose.txt
PASS 3: Adversarial Verify    → See: prompts/p3_adversarial_verify.txt
PASS 4: Risk Synthesis        → See: prompts/p4_risk_synthesis.txt

---

## What to feed each pass

### Pass 1 — one call per image

Input to model:
  - System: contents of p1_image_forensics.txt
  - User: [the image file, base64 or URL] + "Analyze this image."

Output: JSON (image forensics result)
Store as: image_results[image_id]

### Pass 2 — one call per claim (text only)

Input to model:
  - System: contents of p2_claim_decompose.txt
  - User:
    """
    USER CLAIM CONVERSATION:
    {row['user_claim']}

    CLAIM OBJECT: {row['claim_object']}

    EVIDENCE REQUIREMENT:
    requirement_id: {matched_requirement['requirement_id']}
    applies_to: {matched_requirement['applies_to']}
    minimum_image_evidence: {matched_requirement['minimum_image_evidence']}
    """

Output: JSON (claim decomposition result)

### Pass 3 — one call per claim (text only, receives Pass 1 + Pass 2 outputs)

Input to model:
  - System: contents of p3_adversarial_verify.txt
  - User:
    """
    CLAIM SUMMARY (from Pass 2):
    claimed_issue_type: {p2['claimed_issue_type']}
    claimed_object_part: {p2['claimed_object_part']}
    claim_summary: {p2['claim_summary']}
    evidence_rule: {p2['evidence_rule_text']}
    minimum_evidence_checklist: {p2['minimum_evidence_checklist']}
    severity_language: {p2['severity_language']}
    red_flags_in_text: {p2['red_flags_in_text']}

    IMAGE ANALYSIS (from Pass 1):
    {for each image_id, image_result:}
    [image_id: {image_id}]
    object_detected: {image_result['object_detected']}
    object_part_visible: {image_result['object_part_visible']}
    damage_visible: {image_result['damage_visible']}
    damage_types_observed: {image_result['damage_types_observed']}
    image_quality: {image_result['image_quality']}
    severity_estimate: {image_result['severity_estimate']}
    visual_description: {image_result['visual_description']}
    confidence: {image_result['confidence']}

    USER HISTORY:
    past_claim_count: {history['past_claim_count']}
    rejected_claim: {history['rejected_claim']}
    last_90_days_claim_count: {history['last_90_days_claim_count']}
    history_flags: {history['history_flags']}
    history_summary: {history['history_summary']}
    """

Output: JSON (adversarial verdict)

### Pass 4 — one call per claim (text only, receives all prior outputs)

Input to model:
  - System: contents of p4_risk_synthesis.txt
  - User:
    """
    PASS 1 IMAGE ANALYSIS:
    {json.dumps(image_results, indent=2)}

    PASS 2 CLAIM EXTRACTION:
    {json.dumps(p2_output, indent=2)}

    PASS 3 VERDICT:
    {json.dumps(p3_output, indent=2)}

    USER HISTORY:
    user_id: {row['user_id']}
    past_claim_count: {history['past_claim_count']}
    accept_claim: {history['accept_claim']}
    manual_review_claim: {history['manual_review_claim']}
    rejected_claim: {history['rejected_claim']}
    last_90_days_claim_count: {history['last_90_days_claim_count']}
    history_flags: {history['history_flags']}
    history_summary: {history['history_summary']}
    """

Output: JSON → maps directly to output.csv columns

## Evidence requirement matching logic

Match the evidence_requirements.csv row to the claim using:

1. claim_object == requirement claim_object (or 'all')
2. claimed_issue_type (from Pass 2) matches applies_to issue family

Issue family mapping:
- dent → "dent or scratch"
- scratch → "dent or scratch"
- crack → "crack or glass damage"
- glass_shatter → "crack or glass damage"
- broken_part → "broken or missing part"
- missing_part → "broken or missing part"
- torn_packaging → "torn or opened packaging"
- crushed_packaging → "crush or structural damage"
- water_damage → "water or stain damage"
- stain → "water or stain damage"

If no match found, use the 'all' object row if it exists, or omit requirement.

## Model and API settings

Model: claude-sonnet-4-6 (or claude-opus-4-6 for Pass 3 if budget allows)
Max tokens: 1024 (Pass 1, 2, 4) | 1500 (Pass 3 — needs room for reasoning)
Temperature: 0 (deterministic, required for reproducibility)

Pass 1: vision=True (multimodal)
Pass 2: vision=False (text only)
Pass 3: vision=False (text only — receives structured image analysis, not raw images)
Pass 4: vision=False (text only)

---

## Concurrency and rate limit strategy

Pass 1: asyncio.gather — fire all image calls in parallel per claim.
         Between claims: add 0.5s sleep to avoid RPM burst.
Pass 2, 3, 4: sequential per claim. Total ~4 API calls per claim.

For N claims:
  Total API calls ≈ N × (avg_images_per_claim + 3)
  Estimated for 20 claims with 1.5 avg images ≈ 90 calls

Retry strategy: exponential backoff on 429 (rate limit) and 529 (overload).
  Attempt 1: immediate
  Attempt 2: wait 2s
  Attempt 3: wait 8s
  Attempt 4: wait 30s
  After 4 failures: log error, write claim_status=not_enough_information with risk_flag=manual_review_required

---

## JSON parsing safety

Every model response should be parsed with:
  1. Strip any markdown code fences (```json ... ```) 
  2. json.loads()
  3. On failure: retry once with "Output only valid JSON, no markdown"
  4. On second failure: use fallback defaults and flag for manual review

Fallback defaults on parse failure:
  claim_status: not_enough_information
  evidence_standard_met: false
  risk_flags: manual_review_required
  severity: unknown
  valid_image: false

## Evaluation (sample_claims.csv)

Run the full pipeline on sample_claims.csv and compare output to expected columns.
Primary metrics:
  - claim_status accuracy (exact match)
  - evidence_standard_met accuracy (exact match)
  - severity accuracy (exact match)
  - risk_flags recall (were all expected flags captured?)

Secondary metrics:
  - claim_status confusion matrix (3x3: supported / contradicted / not_enough_information)
  - Per-class precision and recall
  - Error analysis: which cases did the pipeline get wrong, and why?

The hardest boundary to get right: contradicted vs not_enough_information.
The adversarial check in Pass 3 is specifically designed to improve this.
